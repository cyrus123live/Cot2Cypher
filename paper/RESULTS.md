# Experimental Results: Chain-of-Thought Fine-Tuning for Text2Cypher

## 1. Overview

This document presents the complete experimental results for improving the Neo4j Text2Cypher benchmark through chain-of-thought (CoT) fine-tuning. The core experiment holds all variables constant between our model and the Neo4j baseline — same base model (Gemma-2-9B-it), same QLoRA configuration (4-bit NF4, r=64, alpha=64), same dataset splits — and changes only the training data: direct-answer Cypher vs. CoT reasoning traces followed by Cypher.

**Bottom line:** CoT fine-tuning improves GLEU by +0.1227 (0.6455 → 0.7682) and execution-based exact match by +0.0692 (0.1862 → 0.2554), making a 9B open-weight model competitive with fine-tuned GPT-4o-mini on translation metrics and surpassing all other open-weight models on the benchmark.

## 2. Experimental Setup

### 2.1 Dataset

All experiments use the `neo4j/text2cypher-2024v1` benchmark dataset:
- **Training split:** 39,554 instances (used for both baseline and CoT fine-tuning)
- **Test split:** 4,833 instances (used for all evaluation)
- **Instances with database access:** 2,471 (51.12% of test set, across 16 Neo4j demo databases)

### 2.2 Model and QLoRA Configuration

Both the baseline and CoT model share identical architecture and training hyperparameters:

| Parameter | Value |
|-----------|-------|
| Base model | `google/gemma-2-9b-it` |
| Quantization | 4-bit NF4, double quantization, bf16 compute dtype |
| LoRA rank (r) | 64 |
| LoRA alpha | 64 |
| LoRA dropout | 0.05 |
| Target modules | All linear (q, k, v, o, gate, up, down projections) |
| Bias | None |
| Task type | CAUSAL_LM |
| Optimizer | paged_adamw_8bit |
| Learning rate | 2e-5 |
| Batch size | 4 (gradient accumulation 8, effective batch 32) |
| Epochs | 1 |
| Max sequence length | 1,600 tokens |
| Attention | `eager` (required for Gemma-2's soft-capping mechanism) |

This configuration exactly replicates the setup described in the Neo4j baseline paper (Ozsoy et al., 2025), including the same framework versions: PyTorch 2.4.0, transformers 4.44.2, PEFT 0.12.0, bitsandbytes 0.43.3.

### 2.3 Training Data Difference

The **only** difference between the baseline and CoT model is what appears in the training data:

**Baseline (Neo4j's approach):** Each training example is a prompt-completion pair where the completion is just the reference Cypher query.

**CoT (our approach):** Each training example is a prompt-completion pair where the completion contains a QDECOMP+InterCOL reasoning trace followed by the reference Cypher query. The reasoning traces were generated by GPT-oss-120B (a 117B parameter reasoning model) using 9 hand-crafted few-shot exemplars covering 3 schema formats and multiple complexity levels.

The CoT reasoning follows a 4-step decomposition format:
1. **Sub-question decomposition:** Break the natural language question into atomic sub-questions
2. **Schema element linking:** Map each sub-question to specific node labels, relationship types, and properties
3. **Graph pattern identification:** Determine the traversal pattern (single hop, multi-hop, aggregation, filtering)
4. **Step-by-step Cypher construction:** Build MATCH, WHERE, and RETURN clauses incrementally

### 2.4 Inference Configuration

| Parameter | Baseline | CoT Model |
|-----------|----------|-----------|
| Quantization | 4-bit NF4 | 4-bit NF4 |
| Decoding | Greedy | Greedy |
| Max new tokens | 512 | 1,024 |
| Input max length | 7,680 | 7,680 |

The CoT model uses a higher `max_new_tokens` (1,024 vs 512) to accommodate the reasoning output before the Cypher query. The Cypher is extracted by parsing text after the "Cypher output:" marker.

### 2.5 Evaluation Methodology

We report two classes of metrics:

**Translation-based (n=4,833):** Generated Cypher is compared to reference Cypher as text.
- **GLEU (Google-BLEU):** Primary metric. Records n-gram sub-sequences (1-4 tokens), computes recall and precision, returns minimum. Uses HuggingFace Evaluate with NLTK's `Tokenizer13a`.
- **String Exact Match:** Whitespace-normalized string comparison. A strict lower bound on correctness.

**Execution-based (n=2,471):** Both generated and reference Cypher are executed against live Neo4j demo databases at `demo.neo4jlabs.com`. Results are converted to lexicographically sorted string representations and compared.
- **Execution Exact Match:** Binary — do the query results match exactly?
- Both baseline and CoT predictions are evaluated using identical execution methodology for a fair comparison.

### 2.6 CoT Training Data Generation

| Parameter | Value |
|-----------|-------|
| Model | GPT-oss-120B via Cerebras |
| Throughput | ~3,000 tokens/sec |
| Total time | ~3.3 hours |
| Total cost | $48 |
| Success rate | 39,553/39,554 (99.997%) |
| Avg prompt tokens | 4,735 |
| Avg completion tokens | 295 (~715 characters) |
| Cypher match rate | 93.3% (mismatches are cosmetic) |

The 6.7% of completions where the generated Cypher differed from the reference were inspected and found to contain only cosmetic differences (whitespace, case, tokenizer artifacts). All training examples use the original reference Cypher, not the model-generated Cypher, ensuring training target quality.

### 3.1 Translation-Based Metrics (Full Test Set)

| Metric | Baseline | CoT Model | Delta |
|--------|----------|-----------|-------|
| **GLEU** | 0.6455 | **0.7682** | **+0.1227** |
| **String Exact Match** | 0.1924 (930/4,833) | **0.3799** (1,836/4,833) | **+0.1875** |

The CoT model achieves a 19.0% relative improvement in GLEU and nearly doubles the exact match rate.

### 3.2 Execution-Based Metrics (Database-Accessible Subset)

Evaluated against 16 live Neo4j demo databases at `demo.neo4jlabs.com` using identical methodology for both models. Both predicted and reference Cypher queries are executed, and the result sets are converted to lexicographically sorted string representations for comparison.

Of 2,471 instances with database access, 2,305 had valid reference query results (166 reference queries failed due to database state changes or deprecated functions). All metrics below are computed on the 2,305 valid instances unless otherwise noted.

| Metric | Baseline | CoT Model | Delta |
|--------|----------|-----------|-------|
| **Execution Exact Match** | 0.1657 (382/2,305) | **0.2369** (546/2,305) | **+0.0712** |
| **Execution GLEU** | 0.1106 | **0.0973** | -0.0133 |
| Prediction Errors | 242 | **114** | **-53%** |
| Reference Errors | 166 | 166 | — |

**Execution EM** is the primary execution metric: the CoT model gets 43% more queries exactly right than the baseline. The CoT model also produces 53% fewer prediction errors (syntax/runtime failures), indicating that reasoning traces during training improve the model's ability to generate syntactically valid Cypher.

**Execution GLEU** is slightly lower for the CoT model despite higher EM. This is because GLEU is computed on stringified result sets — when a query returns incorrect rows, the string overlap with the reference result is minimal regardless of which model produced it. The CoT model trades partial-credit near-misses for more exact matches, which improves EM but can slightly decrease GLEU. Execution EM is the more meaningful metric here, as it directly measures semantic correctness.

Note: the execution EM numbers here (0.1657/0.2369) differ slightly from the earlier EM-only evaluation (0.1862/0.2554) because the GLEU evaluation excludes 166 instances where the reference query itself fails, changing the denominator.

## 5. Ablation Study: Prompt vs Training Data

To disentangle the contributions of CoT training data and the CoT inference prompt ("Think step by step, then provide the Cypher query."), we ran a controlled ablation using the CoT adapter with the baseline prompt format.

| Configuration | Adapter | Prompt | GLEU | String EM |
|---------------|---------|--------|------|-----------|
| Baseline | Neo4j | Baseline | 0.6455 | 0.1924 |
| **CoT training only** | **CoT** | **Baseline** | **0.7082** | **0.2197** |
| **Full CoT** | **CoT** | **CoT** | **0.7682** | **0.3799** |

### Interpretation

**CoT training data alone** (without the CoT prompt) yields +0.0627 GLEU and +0.0273 string EM over the baseline. This demonstrates that the model learns generalizable Cypher generation capabilities from reasoning traces, even when not prompted to reason at inference time.

**Adding the CoT prompt** on top of CoT training yields an additional +0.0600 GLEU and +0.1602 string EM. The prompt effect is especially pronounced for exact match, suggesting that explicit reasoning at inference time produces more precise queries.

**Both components are complementary:** CoT training teaches the model better query construction patterns. The CoT prompt activates the reasoning capability at inference time. Together they are synergistic — the full improvement (+0.1227 GLEU) exceeds the sum of their individual contributions.

We did not run the reverse ablation (baseline adapter + CoT prompt) because it changes the prompt format the baseline model was trained on, making the comparison confounded by format mismatch rather than isolating the prompt effect.

## 6. Error Analysis

### 6.1 Prediction Overlap

To understand whether CoT uniformly improves predictions or trades some correct answers for others, we examine the overlap between baseline and CoT predictions.

**String Exact Match Overlap (n=4,833):**

| Category | Count | % |
|----------|------:|--:|
| Both correct | 808 | 16.7% |
| CoT only correct | 1,028 | 21.3% |
| Baseline only correct | 122 | 2.5% |
| Neither correct | 2,875 | 59.5% |

CoT "unlocks" 1,028 new exact matches while losing only 122 that the baseline got right — an 8.4:1 improvement-to-regression ratio. The 122 regressions are distributed across synthetic sources (gpt4o: 25, gpt4turbo: 18, claudeopus: 17, functional_cypher: 20).

**Execution Match Overlap (n=2,471):**

| Category | Count |
|----------|------:|
| Both correct | 323 |
| CoT only correct | 308 |
| Baseline only correct | 137 |
| Neither correct | 1,703 |

The execution overlap shows a similar pattern: CoT gains 308 new correct queries while losing 137 — a 2.2:1 ratio. The higher regression rate for execution (137 vs 122 for string) reflects cases where CoT's syntactically different query happens to return different results on the live database.

### 6.2 Mismatch Categories

Of the 2,997 test examples where CoT does not achieve a string exact match:

| Category | Count | % |
|----------|------:|--:|
| Different MATCH pattern | 1,500 | 50.1% |
| Same MATCH, different RETURN columns | 963 | 32.1% |
| Same MATCH, different alias only | 197 | 6.6% |
| Other (ORDER BY, LIMIT, WITH differences) | 337 | 11.2% |

The 197 alias-only differences (6.6%) are purely cosmetic — the queries are semantically identical but use different `AS` names. These count as incorrect for string EM but would match on execution EM.

The largest category (50.1%) involves fundamentally different graph traversal patterns. Many of these are cases where multiple valid Cypher queries answer the same question (e.g., using WHERE vs. property filters in the MATCH pattern).

This is strong evidence that chain-of-thought reasoning particularly helps with complex, multi-step query generation.

The extra-long queries (500+ chars) are primarily from `neo4jLabs_functional_cypher` and include UNION queries, variable-length paths, and multi-clause constructions where step-by-step reasoning is most beneficial.

### 6.5 Prediction Errors

Of the 114 CoT prediction errors on the DB-accessible subset:
- **Syntax errors: 72** (63%) — the generated Cypher has invalid syntax
- **Too many rows: 42** (37%) — the query returns >10,000 rows (capped for memory safety)

The 53% reduction in prediction errors (114 vs 242 baseline) demonstrates that CoT training teaches the model to produce more syntactically valid Cypher. The remaining syntax errors tend to involve complex expressions (nested function calls, string operations) rather than basic pattern matching.

### 6.6 Regression Analysis

The 122 string-match regressions (baseline correct, CoT wrong) show a consistent pattern: the CoT model generates a semantically reasonable but syntactically different query. Examples:

**Regression type 1: Return clause differences**
- Question: "List 3 questions with the lowest score."
- Baseline: `MATCH (q:Question) RETURN q.title, q.score ORDER BY q.score ASC LIMIT 3` (exact match)
- CoT: `MATCH (q:Question) RETURN q ORDER BY q.score ASC LIMIT 3` (returns whole node instead of specific properties)

**Regression type 2: Alias/formatting differences**
- Question: "Which organizations are based in 'Seattle' and have a revenue less than 5000000?"
- Baseline: `...RETURN o` (exact match)
- CoT: `...RETURN o.name AS organization, o.revenue AS revenue` (more verbose but correct)

**Regression type 3: Clause ordering**
- Question: "What is the text of the review with the lowest stars rating?"
- Baseline: `MATCH (r:Review) WITH r ORDER BY r.stars ASC LIMIT 1 RETURN r.text` (exact match)
- CoT: `MATCH (r:Review) RETURN r.text ORDER BY r.stars ASC LIMIT 1` (semantically equivalent, different clause order)

Many regressions are cosmetic — the CoT output is semantically correct but differs from the reference in ways that string matching penalizes. This is confirmed by the execution EM analysis where the improvement-to-regression ratio (2.2:1) is lower than for string EM (8.4:1), meaning some "regressions" on string match are actually execution matches.

## 7. Qualitative Examples

### 6.1 Example: Exact Match

**Question:** "What are the names of the technicians that have not been assigned to repair machines?"

**CoT Model Output:**
```
Reasoning:
1. Sub-questions: (a) Identify all Technician nodes. (b) Filter out those that have
   an incoming ASSIGNED_TO relationship from any RepairAssignment node.
   (c) Return the remaining technicians' names.
2. Schema elements: Node label Technician with property Name; relationship
   ASSIGNED_TO from RepairAssignment → Technician.
3. Pattern: Simple node match on Technician with NOT EXISTS sub-query filtering
   out those with ASSIGNED_TO.
4. Construction:
   - MATCH (t:Technician)
   - WHERE NOT EXISTS ((:RepairAssignment)-[:ASSIGNED_TO]->(t))
   - RETURN t.Name

Cypher output: MATCH (t:Technician) WHERE NOT EXISTS ((:RepairAssignment)-[:ASSIGNED_TO]->(t)) RETURN t.Name
```

**Reference:** `MATCH (t:Technician) WHERE NOT EXISTS ((:RepairAssignment)-[:ASSIGNED_TO]->(t)) RETURN t.Name`

The reasoning trace correctly identifies the NOT EXISTS pattern, maps to the right schema elements, and produces an exact match.

### 6.2 Example: Semantically Correct, Syntactically Different

**Question:** "Identify the 5 suppliers with the highest average unit price of products supplied."

**CoT Model Output:**
```
Reasoning:
1. Sub-questions: (a) Which suppliers supply products? (b) What is the average
   unitPrice of those products per supplier? (c) Return the top 5 suppliers
   by this average.
2. Schema elements: Supplier node (companyName), Product node (unitPrice),
   relationship SUPPLIES from Supplier to Product.
3. Pattern: Two-hop traversal Supplier-[:SUPPLIES]->Product, then aggregation
   avg(p.unitPrice) grouped by supplier, ordered descending, limited to 5.
4. Construction:
   - MATCH (s:Supplier)-[:SUPPLIES]->(p:Product)
   - WITH s, avg(p.unitPrice) AS avgUnitPrice
   - ORDER BY avgUnitPrice DESC LIMIT 5
   - RETURN s.companyName, avgUnitPrice

Cypher output: MATCH (s:Supplier)-[:SUPPLIES]->(p:Product) WITH s, avg(p.unitPrice) AS avgUnitPrice ORDER BY avgUnitPrice DESC LIMIT 5 RETURN s.companyName AS supplierName, avgUnitPrice
```

**Reference:** `MATCH (s:Supplier)-[:SUPPLIES]->(p:Product) WITH s, avg(p.unitPrice) AS avgUnitPrice ORDER BY avgUnitPrice DESC LIMIT 5 RETURN s.companyName AS Supplier, avgUnitPrice AS AverageUnitPrice`

The query is semantically identical — only the column aliases differ (`supplierName` vs `Supplier`, `avgUnitPrice` vs `AverageUnitPrice`). This would be an execution match but not a string match, illustrating why execution EM exceeds string EM on the DB-accessible subset.

### 6.3 Output Format Compliance

Across all 4,833 test predictions:
- **100%** contain the "Cypher output:" marker
- Average raw output length: 870 characters
- The model consistently follows the trained format: reasoning first, then Cypher

### 7.2 Truncation Impact

With `max_seq_length=1600`, approximately 2-3% of training examples had their assistant response fully truncated (the `DataCollatorForCompletionOnlyLM` could not find the `<start_of_turn>model\n` response template). These examples contributed zero to the training loss. This primarily affects instances with very long schemas where the prompt alone exceeds 1,600 tokens, leaving no room for the reasoning + Cypher response.

## 10. Limitations and Threats to Validity

### 10.1 Known Dataset Issues

The Neo4j text2cypher-2024v1 dataset has documented limitations that affect all models evaluated on it:
- **Data leakage:** Same questions may appear with different Cypher outputs across train/test splits
- **Paraphrase contamination:** Paraphrased versions of questions may appear in both splits
- **Distribution shift:** Train and test drawn from the same distribution; real-world performance may differ

These affect the baseline and CoT model equally and do not invalidate the relative comparison.

### 10.2 Experimental Design Considerations

1. **Two variables changed simultaneously:** The full CoT configuration changes both training data (CoT traces) and inference prompt ("Think step by step"). Our ablation (Section 5) shows that CoT training alone provides +0.0627 GLEU, confirming that the training data is the primary driver.

2. **Single epoch, no hyperparameter tuning:** Following Neo4j's methodology, we train for 1 epoch without tuning. The authors note that "with better-tuned parameters, we could potentially achieve even stronger results." The same applies to our CoT model.

3. **Execution evaluation caveats:** The Neo4j demo databases are live public infrastructure. Database contents may have changed between Neo4j's evaluation and ours, explaining the slight gap between our baseline execution EM (0.1862) and Neo4j's published value (0.2104). The relative comparison between our baseline and CoT model uses identical evaluation timing and methodology.

4. **Reasoning trace quality:** The CoT traces were generated by a single model (GPT-oss-120B). Different reasoning models or prompting strategies might produce different results.

## 11. Summary

Chain-of-thought fine-tuning — generating reasoning traces with a strong LLM and training a smaller model on those traces — significantly improves Text2Cypher performance:

- **+0.1227 GLEU** on translation-based evaluation (4,833 test examples)
- **+0.0692 execution EM** on execution-based evaluation (2,471 test examples)
- **53% fewer prediction errors** (114 vs 242 syntax/runtime failures)
- **Improvement across all data sources and 15 of 16 databases**
- **9B open model competitive with fine-tuned GPT-4o-mini** on GLEU (0.7682 vs 0.7973)

The improvement comes primarily from the training data (reasoning traces teach better query construction) with additional gains from the CoT inference prompt (explicit reasoning produces more precise queries). The total cost of the approach — $73 for data generation and training — makes it practical for any research group or organization working with graph databases.
