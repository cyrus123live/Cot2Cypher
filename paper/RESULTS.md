# Experimental Results: Chain-of-Thought Fine-Tuning for Text2Cypher

## 1. Overview

This document presents the complete experimental results for improving the Neo4j Text2Cypher benchmark through chain-of-thought (CoT) fine-tuning. The core experiment holds all variables constant between our model and the Neo4j baseline — same base model (Gemma-2-9B-it), same QLoRA configuration (4-bit NF4, r=64, alpha=64), same dataset splits — and changes only the training data: direct-answer Cypher vs. CoT reasoning traces followed by Cypher.

**Bottom line:** CoT fine-tuning improves GLEU by +0.1227 (0.6455 → 0.7682) and execution-based exact match by +0.0692 (0.1862 → 0.2554), making a 9B open-weight model competitive with fine-tuned GPT-4o-mini on translation metrics and surpassing all other open-weight models on the benchmark.

## 2. Experimental Setup

### 2.1 Dataset

All experiments use the `neo4j/text2cypher-2024v1` benchmark dataset:
- **Training split:** 39,554 instances (used for both baseline and CoT fine-tuning)
- **Test split:** 4,833 instances (used for all evaluation)
- **Instances with database access:** 2,471 (51.12% of test set, across 16 Neo4j demo databases)

### 2.2 Model and QLoRA Configuration

Both the baseline and CoT model share identical architecture and training hyperparameters:

| Parameter | Value |
|-----------|-------|
| Base model | `google/gemma-2-9b-it` |
| Quantization | 4-bit NF4, double quantization, bf16 compute dtype |
| LoRA rank (r) | 64 |
| LoRA alpha | 64 |
| LoRA dropout | 0.05 |
| Target modules | All linear (q, k, v, o, gate, up, down projections) |
| Bias | None |
| Task type | CAUSAL_LM |
| Optimizer | paged_adamw_8bit |
| Learning rate | 2e-5 |
| Batch size | 4 (gradient accumulation 8, effective batch 32) |
| Epochs | 1 |
| Max sequence length | 1,600 tokens |
| Attention | `eager` (required for Gemma-2's soft-capping mechanism) |

This configuration exactly replicates the setup described in the Neo4j baseline paper (Ozsoy et al., 2025), including the same framework versions: PyTorch 2.4.0, transformers 4.44.2, PEFT 0.12.0, bitsandbytes 0.43.3.

### 2.3 Training Data Difference

The **only** difference between the baseline and CoT model is what appears in the training data:

**Baseline (Neo4j's approach):** Each training example is a prompt-completion pair where the completion is just the reference Cypher query.

**CoT (our approach):** Each training example is a prompt-completion pair where the completion contains a QDECOMP+InterCOL reasoning trace followed by the reference Cypher query. The reasoning traces were generated by GPT-oss-120B (a 117B parameter reasoning model) using 9 hand-crafted few-shot exemplars covering 3 schema formats and multiple complexity levels.

The CoT reasoning follows a 4-step decomposition format:
1. **Sub-question decomposition:** Break the natural language question into atomic sub-questions
2. **Schema element linking:** Map each sub-question to specific node labels, relationship types, and properties
3. **Graph pattern identification:** Determine the traversal pattern (single hop, multi-hop, aggregation, filtering)
4. **Step-by-step Cypher construction:** Build MATCH, WHERE, and RETURN clauses incrementally

### 2.4 Inference Configuration

| Parameter | Baseline | CoT Model |
|-----------|----------|-----------|
| Quantization | 4-bit NF4 | 4-bit NF4 |
| Decoding | Greedy | Greedy |
| Max new tokens | 512 | 1,024 |
| Input max length | 7,680 | 7,680 |

The CoT model uses a higher `max_new_tokens` (1,024 vs 512) to accommodate the reasoning output before the Cypher query. The Cypher is extracted by parsing text after the "Cypher output:" marker.

### 2.5 Evaluation Methodology

We report two classes of metrics:

**Translation-based (n=4,833):** Generated Cypher is compared to reference Cypher as text.
- **GLEU (Google-BLEU):** Primary metric. Records n-gram sub-sequences (1-4 tokens), computes recall and precision, returns minimum. Uses HuggingFace Evaluate with NLTK's `Tokenizer13a`.
- **String Exact Match:** Whitespace-normalized string comparison. A strict lower bound on correctness.

**Execution-based (n=2,471):** Both generated and reference Cypher are executed against live Neo4j demo databases at `demo.neo4jlabs.com`. Results are converted to lexicographically sorted string representations and compared.
- **Execution Exact Match:** Binary — do the query results match exactly?
- Both baseline and CoT predictions are evaluated using identical execution methodology for a fair comparison.

### 2.6 CoT Training Data Generation

| Parameter | Value |
|-----------|-------|
| Model | GPT-oss-120B via Cerebras |
| Throughput | ~3,000 tokens/sec |
| Total time | ~3.3 hours |
| Total cost | $48 |
| Success rate | 39,553/39,554 (99.997%) |
| Avg prompt tokens | 4,735 |
| Avg completion tokens | 295 (~715 characters) |
| Cypher match rate | 93.3% (mismatches are cosmetic) |

The 6.7% of completions where the generated Cypher differed from the reference were inspected and found to contain only cosmetic differences (whitespace, case, tokenizer artifacts). All training examples use the original reference Cypher, not the model-generated Cypher, ensuring training target quality.

The CoT model achieves a 19.0% relative improvement in GLEU and nearly doubles the exact match rate.

### 3.2 Execution-Based Metrics (Database-Accessible Subset)

Evaluated against 16 live Neo4j demo databases at `demo.neo4jlabs.com` using identical methodology for both models. Both predicted and reference Cypher queries are executed, and the result sets are converted to lexicographically sorted string representations for comparison.

Of 2,471 instances with database access, 2,305 had valid reference query results (166 reference queries failed due to database state changes or deprecated functions). All metrics below are computed on the 2,305 valid instances unless otherwise noted.

| Metric | Baseline | CoT Model | Delta |
|--------|----------|-----------|-------|
| **Execution Exact Match** | 0.1657 (382/2,305) | **0.2369** (546/2,305) | **+0.0712** |
| **Execution GLEU** | 0.1106 | **0.0973** | -0.0133 |
| Prediction Errors | 242 | **114** | **-53%** |
| Reference Errors | 166 | 166 | — |

**Execution EM** is the primary execution metric: the CoT model gets 43% more queries exactly right than the baseline. The CoT model also produces 53% fewer prediction errors (syntax/runtime failures), indicating that reasoning traces during training improve the model's ability to generate syntactically valid Cypher.

**Execution GLEU** is slightly lower for the CoT model despite higher EM. This is because GLEU is computed on stringified result sets — when a query returns incorrect rows, the string overlap with the reference result is minimal regardless of which model produced it. The CoT model trades partial-credit near-misses for more exact matches, which improves EM but can slightly decrease GLEU. Execution EM is the more meaningful metric here, as it directly measures semantic correctness.

Note: the execution EM numbers here (0.1657/0.2369) differ slightly from the earlier EM-only evaluation (0.1862/0.2554) because the GLEU evaluation excludes 166 instances where the reference query itself fails, changing the denominator.

## 5. Ablation Study: Prompt vs Training Data

To disentangle the contributions of CoT training data and the CoT inference prompt ("Think step by step, then provide the Cypher query."), we ran a controlled ablation using the CoT adapter with the baseline prompt format.

| Configuration | Adapter | Prompt | GLEU | String EM |
|---------------|---------|--------|------|-----------|
| Baseline | Neo4j | Baseline | 0.6455 | 0.1924 |
| **CoT training only** | **CoT** | **Baseline** | **0.7082** | **0.2197** |
| **Full CoT** | **CoT** | **CoT** | **0.7682** | **0.3799** |

### Interpretation

**CoT training data alone** (without the CoT prompt) yields +0.0627 GLEU and +0.0273 string EM over the baseline. This demonstrates that the model learns generalizable Cypher generation capabilities from reasoning traces, even when not prompted to reason at inference time.

**Adding the CoT prompt** on top of CoT training yields an additional +0.0600 GLEU and +0.1602 string EM. The prompt effect is especially pronounced for exact match, suggesting that explicit reasoning at inference time produces more precise queries.

**Both components are complementary:** CoT training teaches the model better query construction patterns. The CoT prompt activates the reasoning capability at inference time. Together they are synergistic — the full improvement (+0.1227 GLEU) exceeds the sum of their individual contributions.

We did not run the reverse ablation (baseline adapter + CoT prompt) because it changes the prompt format the baseline model was trained on, making the comparison confounded by format mismatch rather than isolating the prompt effect.

### 6.1 Prediction Overlap

To understand whether CoT uniformly improves predictions or trades some correct answers for others, we examine the overlap between baseline and CoT predictions.

**String Exact Match Overlap (n=4,833):**

| Category | Count | % |
|----------|------:|--:|
| Both correct | 808 | 16.7% |
| CoT only correct | 1,028 | 21.3% |
| Baseline only correct | 122 | 2.5% |
| Neither correct | 2,875 | 59.5% |

CoT "unlocks" 1,028 new exact matches while losing only 122 that the baseline got right — an 8.4:1 improvement-to-regression ratio. The 122 regressions are distributed across synthetic sources (gpt4o: 25, gpt4turbo: 18, claudeopus: 17, functional_cypher: 20).

**Execution Match Overlap (n=2,471):**

| Category | Count |
|----------|------:|
| Both correct | 323 |
| CoT only correct | 308 |
| Baseline only correct | 137 |
| Neither correct | 1,703 |

The execution overlap shows a similar pattern: CoT gains 308 new correct queries while losing 137 — a 2.2:1 ratio. The higher regression rate for execution (137 vs 122 for string) reflects cases where CoT's syntactically different query happens to return different results on the live database.

### 6.2 Mismatch Categories

Of the 2,997 test examples where CoT does not achieve a string exact match:

| Category | Count | % |
|----------|------:|--:|
| Different MATCH pattern | 1,500 | 50.1% |
| Same MATCH, different RETURN columns | 963 | 32.1% |

The 197 alias-only differences (6.6%) are purely cosmetic — the queries are semantically identical but use different `AS` names. These count as incorrect for string EM but would match on execution EM.

The largest category (50.1%) involves fundamentally different graph traversal patterns. Many of these are cases where multiple valid Cypher queries answer the same question (e.g., using WHERE vs. property filters in the MATCH pattern).

### 6.5 Prediction Errors

Of the 114 CoT prediction errors on the DB-accessible subset:
- **Syntax errors: 72** (63%) — the generated Cypher has invalid syntax
- **Too many rows: 42** (37%) — the query returns >10,000 rows (capped for memory safety)

### 6.6 Regression Analysis

The 122 string-match regressions (baseline correct, CoT wrong) show a consistent pattern: the CoT model generates a semantically reasonable but syntactically different query. Examples:

**Regression type 1: Return clause differences**
- Question: "List 3 questions with the lowest score."
- Baseline: `MATCH (q:Question) RETURN q.title, q.score ORDER BY q.score ASC LIMIT 3` (exact match)
- CoT: `MATCH (q:Question) RETURN q ORDER BY q.score ASC LIMIT 3` (returns whole node instead of specific properties)

**Regression type 2: Alias/formatting differences**
- Question: "Which organizations are based in 'Seattle' and have a revenue less than 5000000?"
- Baseline: `...RETURN o` (exact match)
- CoT: `...RETURN o.name AS organization, o.revenue AS revenue` (more verbose but correct)

**Regression type 3: Clause ordering**
- Question: "What is the text of the review with the lowest stars rating?"
- Baseline: `MATCH (r:Review) WITH r ORDER BY r.stars ASC LIMIT 1 RETURN r.text` (exact match)
- CoT: `MATCH (r:Review) RETURN r.text ORDER BY r.stars ASC LIMIT 1` (semantically equivalent, different clause order)

Many regressions are cosmetic — the CoT output is semantically correct but differs from the reference in ways that string matching penalizes. This is confirmed by the execution EM analysis where the improvement-to-regression ratio (2.2:1) is lower than for string EM (8.4:1), meaning some "regressions" on string match are actually execution matches.

### 6.7 The 1-Hop Paradox: Schema Grounding as Constraint Satisfaction

A surprising finding in the per-feature analysis is that the baseline performs *worst* on the *simplest* traversals:

| Traversal | Baseline EM | CoT EM | Delta |
|-----------|:-----------:|:------:|:-----:|
| 0-hop (node only) | 0.272 | 0.456 | +0.183 |
| 1-hop | 0.023 | 0.437 | **+0.414** |
| 2-hop | 0.197 | 0.361 | +0.164 |
| 3+-hop | 0.075 | 0.212 | +0.137 |

The baseline achieves only 2.3% EM on 1-hop queries, worse than 2-hop (19.7%) or even 3+-hop (7.5%). CoT's largest improvement is correspondingly on 1-hop (+0.414), not the structurally harder categories.

**This paradox holds within the same data source.** Restricting to `neo4jLabs_functional_cypher` (eliminating cross-source confounds):

| Traversal | n | Baseline EM | CoT EM | Delta |
|-----------|--:|:-----------:|:------:|:-----:|
| 0-hop | 535 | 0.493 | 0.877 | +0.383 |
| 1-hop | 186 | 0.038 | 0.887 | **+0.850** |
| 2-hop | 583 | 0.506 | 0.925 | +0.418 |
| 3+-hop | 179 | 0.251 | 0.810 | +0.559 |

The baseline gets 49.3% on 0-hop and 50.6% on 2-hop, but only 3.8% on 1-hop. Why?

**Explanation: 1-hop queries combine traversal with aggregation.** Analyzing feature prevalence within `functional_cypher` by hop count reveals that 1-hop queries disproportionately require WITH clauses (34.4% vs 14.4% for 2-hop), DISTINCT (25.8% vs 17.7%), and COUNT (32.8% vs 27.8%). The baseline can handle traversals (2-hop: 50.6%) and node-only aggregation (0-hop: 49.3%) separately, but fails when both are required together. For 1-hop queries with WITH clauses, the baseline achieves 0.0% EM. For those with DISTINCT, also 0.0%.

**CoT resolves this through explicit compositional reasoning.** The 4-step decomposition first identifies the traversal pattern (step 3), then constructs the aggregation logic separately (step 4), composing them in sequence. This mirrors how compositional generalization works in human reasoning: learn primitives, then learn to compose them. The baseline's direct-answer training never provides signal about how to combine these primitives.

**Schema complexity interaction.** The effect is strongest on simple schemas (0-2 relationship types): +0.77 EM improvement on 1-hop. On complex schemas (11-20 relationship types), the improvement drops to +0.03. This suggests that CoT's schema grounding (step 2: InterCOL) is most effective when the schema is small enough to reason about within the token budget.

#### Schema Ambiguity as a Direct Mechanism

To directly probe schema-grounding behavior, we isolated the 1,119 1-hop queries where both endpoints are explicitly typed and the reference relationship is declared in the schema. For each, we computed *schema ambiguity* — the number of distinct relationship types declared in the schema between the (unordered) pair of node labels involved. Ambiguity ≥ 2 means the model must read the schema to disambiguate; ambiguity = 1 means the relationship type is uniquely determined by the node pair.

We measure two metrics: **string EM** (whole-query lexical match) and **correct relationship type used** (does the predicted Cypher reference the right `:REL_TYPE`?). The latter is a direct test of schema grounding, isolated from surface-form variation.

| Ambiguity | n | Baseline `correct rel` | CoT `correct rel` | Δ |
|-----------|--:|:----------------------:|:-----------------:|:-:|
| =1 (unambiguous) | 658 | 0.948 | 0.970 | +0.021 |
| =2 | 156 | 0.878 | 0.949 | +0.071 |
| =3 | 65 | 0.923 | 0.969 | +0.046 |
| ≥4 | 240 | 0.913 | 0.983 | +0.071 |

**The baseline incurs a 4.6pp ambiguity penalty** (94.8% on unambiguous → 90.2% on ambiguous): when multiple relationship types are valid between the same node pair, the baseline is more likely to pick the wrong one. **CoT eliminates this penalty entirely** (96.96% on both unambiguous and ambiguous, gap = 0.0pp).

This is a clean mechanistic confirmation of the schema-grounding hypothesis: CoT's InterCOL step (linking sub-questions to specific schema elements) forces the model to consult the schema rather than relying on relationship-type priors learned from the training distribution. The result is that the model's ability to pick the right relationship becomes invariant to whether the schema is locally ambiguous or not — exactly the property a schema-grounded learner should have.

Execution-EM tells the same story (DB-eligible 1-hop subset, n=1,096): CoT improvement is +0.080 / +0.086 / +0.092 across ambiguity buckets 1/2/3, modestly favoring the ambiguous case. The 4+ bucket is dominated by social-network schemas where both models likely memorize the canonical relationship type, making the ambiguity test less diagnostic.

### 6.8 Latent vs Active Reasoning: Two Mechanisms of CoT Distillation

Our ablation (Section 5.4) reveals that CoT distillation operates through two separable mechanisms with different cost-accuracy profiles. We frame these as **latent reasoning** (knowledge distilled into the weights, no explicit reasoning at inference) and **active reasoning** (the model generates reasoning before the answer):

| Configuration | GLEU | String EM | Exec EM³ | Out Tokens | $/1k queries¹ | Latency² |
|---------------|:----:|:---------:|:--------:|:----------:|:-------------:|:--------:|
| Baseline (no CoT training, no CoT prompt) | 0.6455 | 0.1924 | 0.1862 | ~39 | $0.36 | 1.3s |
| **Latent** (CoT training, baseline prompt) | 0.7082 | 0.2197 | 0.2153 | ~36 | $0.33 | 1.2s |
| **Active** (CoT training, CoT prompt) | 0.7682 | 0.3799 | 0.2554 | ~247 | $2.29 | 8.2s |

¹ Estimated on a single A10G GPU ($1.00/hr cloud rate, ~30 tok/s for Gemma-2-9B 4-bit).
² Wall-clock per query for the same setup.
³ Execution exact match against `demo.neo4jlabs.com` over the 2,471 DB-eligible test instances.

The three metrics tell different stories about what latent reasoning recovers:

| Metric | Δ Active over Baseline | Δ Latent over Baseline | **Latent recovery** |
|--------|:---------------------:|:----------------------:|:-------------------:|
| GLEU | +0.123 | +0.063 | **51%** |
| Exec EM | +0.069 | +0.029 | **42%** |
| String EM | +0.188 | +0.027 | **15%** |

**Latent reasoning** matches baseline inference cost (~36 output tokens) but recovers about half of the GLEU gain and a substantial 42% of the executable-correctness gain. SFT on reasoning traces teaches better query-construction priors that persist into direct-answer decoding. This is conceptually similar to how rationale-augmented fine-tuning improves models' ability to generate code or proofs without explicitly producing the rationale at inference.

**Active reasoning** is what's required for the canonical surface form: 85% of the exact-match improvement comes from explicit inference-time reasoning. This is also where executable correctness is fully realized — the remaining 58% of the exec-EM gain over baseline. The cost is ~7× more output tokens and ~7× higher latency.

**Practical implication.** The two configurations target different deployment regimes:
- *Search / retrieval / IR-style applications* where Cypher only needs to be approximately right (e.g., as a query candidate to re-rank or as a structured anchor for embedding-based retrieval) can use **latent** at baseline inference cost: GLEU climbs from 0.65 to 0.71 for free.
- *Production database execution* where the query must actually run correctly benefits substantially from **active** despite the 7× cost: exec EM jumps from 0.215 (latent) to 0.255 (active), and string-canonical correctness jumps from 0.22 to 0.38. The marginal active-only gain on exec EM is +0.040, recovering the second half of the executable-correctness improvement.

This cost-accuracy decomposition has not been previously characterized for query generation. The MDPI RL paper (Tran et al., 2025) reports a single number for their model, conflating latent and active reasoning into one.

### 6.9 Graph Reasoning Primitives: A Taxonomy

The per-feature accuracy results (Section 6.4) reveal a principled hierarchy of graph-reasoning operations, ordered by how much they benefit from explicit chain-of-thought. The hierarchy tracks **specification difficulty** — how much structural decision-making the writer must perform when composing the Cypher clause. We emphasize that this is *not* about the computational difficulty of the underlying graph problem: the model is asked to *write* a query that the database engine then executes. A child can type `RETURN count(*)` without being able to count subgraphs. What we are characterizing is how hard it is for the model to *compose the right syntactic structure* for a given semantic intent.

**Tier 1: Pattern composition and structural specification (CoT benefit ≥ +0.30 EM)**
| Operation | Δ EM | Specification challenge |
|-----------|:----:|-------------------------|
| UNION | +0.520 | Construct two structurally distinct patterns that together cover the question's intent |
| 1-hop traversal | +0.414 | Select among competing relationship types that connect the same node pair |
| Variable-length path | +0.321 | Specify a recursive constraint (`*1..k`) and choose its bounds |

These all require the model to decide *what graph pattern to search for*. The pattern has no direct analogue in the natural-language input — it must be inferred from the question's logical structure. CoT's gains here are largest because the four-step decomposition externalizes the pattern-construction step (step 3) that the baseline must otherwise perform implicitly in a single forward pass. (As an aside, the underlying graph problems are also computationally hard — subgraph isomorphism is NP-hard, reachability under bounded length is exponential in the bound — but this is suggestive rather than load-bearing for our claim. The relevant difficulty is at the *writing* layer.)

**Tier 2: Operation choice on top of a matched pattern (CoT benefit +0.15 to +0.25 EM)**
| Operation | Δ EM | Specification challenge |
|-----------|:----:|-------------------------|
| COLLECT | +0.218 | Decide whether results should be aggregated into a list per group |
| EXISTS / NOT EXISTS | +0.208 | Wrap a subpattern in an existence predicate |
| DISTINCT | +0.188 | Decide whether duplicates need elimination |
| WITH | +0.181 | Insert an intermediate stage to bind values for downstream clauses |
| Aggregation (AVG/SUM/MIN/MAX) | +0.165 | Choose the right aggregator and the right grouping |

These operations sit on top of a pattern that has already been matched. Once the pattern is fixed, the writer is choosing among a small set of clauses for a specific purpose. CoT helps because step 4 of the trace breaks down clause assembly explicitly, but the choice space is smaller than for Tier 1.

**Tier 3: Result formatting (CoT benefit ≤ +0.10 EM)**
| Operation | Δ EM | Specification challenge |
|-----------|:----:|-------------------------|
| ORDER BY | +0.088 | Read a sort key off the question (often "top", "highest", "most") |
| LIMIT | +0.078 | Read a number off the question |

These take a result table as input and produce a result table. They touch no graph structure and have a single canonical syntactic form. The baseline already handles them well because they appear in stereotyped positions and the relevant value is usually explicit in the question.

**Interpretation.** Reading the three tiers together, **CoT teaches graph-structural specification, not general carefulness or longer outputs.** The CoT gain on each Cypher feature is well-predicted by how much structural decision-making the writer must perform:
1. Patterns that must be composed / searched / recursively bounded (Tier 1): largest gains, because step 3 of the trace externalizes pattern construction.
2. Clause choice on top of a fixed pattern (Tier 2): medium gains, because step 4 of the trace decomposes clause assembly.
3. Mechanical post-processing read off the question (Tier 3): smallest gains, because no structural reasoning is needed.

This taxonomy makes a falsifiable prediction: a *non-reasoning* fine-tune on the same volume of (question, Cypher) pairs without traces should improve Tier 3 most (memorization helps formatting) and Tier 1 least. Our latent ablation (§6.8) is consistent with this — most of the active-reasoning advantage on exact match comes from Tier 1 features, while Tier 3 is approximately solved by both configurations.

### 6.10 Self-Consistency

We evaluated self-consistency with 5 sampled generations at temperature 0.7, selecting the majority-voted Cypher per example.

| Metric | Greedy | SC@5 | Delta |
|--------|:------:|:----:|:-----:|
| GLEU (n=4,833) | 0.7682 | 0.7634 | -0.0048 |
| String EM (n=4,833) | 0.3799 | **0.3886** | +0.0087 |
| Exec EM (n=2,471) | 0.2554 | 0.2509 | -0.0045 |

Self-consistency provides a modest +0.9% string EM improvement (+50 net correct) but slightly decreases GLEU and execution EM. Candidate-agreement statistics:
- Full agreement (5/5): 25.6% of examples
- Majority agreement (3+/5): 49.8% of examples

The limited gains contrast sharply with text-to-SQL results (CSC-SQL, STaR-SQL report +3-8%). This suggests Cypher's more constrained output space leaves less room for diversity-based voting to filter noise. The model's greedy output is already high-confidence for most queries — supporting our broader finding that the Cypher output space is sufficiently constrained for CoT distillation alone to be near-optimal.

### 6.11 Cross-Benchmark Generalization (ZOGRASCOPE)

We evaluate on ZOGRASCOPE (Cazzaro et al., 2025), a Cypher benchmark on the Pole crime knowledge graph with explicit IID, compositional, and length generalization splits. We report two configurations: **zero-shot** (the Neo4j-trained model with no Pole adaptation) and **fine-tuned** (further QLoRA fine-tuning on 4,324 ZOGRASCOPE training-set traces distilled from GPT-oss-120B via Groq, same procedure as the Neo4j CoT distillation pipeline).

ZOGRASCOPE uses Neo4j 5+ inline-WHERE syntax (e.g., `MATCH (x:Person WHERE x.name = "John")`), while our Neo4j training data used exclusively traditional separate WHERE clauses. 100% of ZOGRASCOPE references use inline syntax; 0% of the original training data does — so the zero-shot evaluation tests both schema-level and syntax-level transfer.

#### Zero-shot results (no Pole training)

| Model | IID | Compositional | Length |
|-------|:---:|:-------------:|:------:|
| Llama 3.2-3B | 3.38% | 1.48% | 0.24% |
| Mistral 7B | 8.85% | 4.82% | 0.56% |
| Qwen3 4B | 10.41% | 7.04% | 0.88% |
| **Our CoT Gemma-2-9B (zero-shot)** | **14.45%** | **6.23%** | **3.35%** |
| GPT-4o | 41.67% | 32.91% | 16.28% |

Best open-weight zero-shot performance, particularly on length generalization (4–14× over other open-weight models <10B). Despite the syntactic dialect mismatch, the model produces functional traditional-Cypher queries that successfully execute against the Pole graph.

#### Fine-tuned results (4,324 ZOGRASCOPE CoT traces, 1 epoch QLoRA)

We further fine-tune the Neo4j-trained CoT model on ZOGRASCOPE training data. Following the same distillation pipeline as before, we generate 4,324 reasoning traces over the ZOGRASCOPE training set via GPT-oss-120B (Groq, 99.8% Cypher match rate, ~$0 since Groq's free tier covers the volume), then apply 1 epoch of QLoRA fine-tuning on the resulting traces. Training: 20 minutes on a single H100. The fine-tuned model also adopts ZOGRASCOPE's inline-WHERE syntactic dialect (100% inline-WHERE, 100% x0/x1/x2 variable naming).

| Model | IID | Compositional | Length |
|-------|:---:|:-------------:|:------:|
| Our CoT Gemma-2-9B (zero-shot) | 14.45% | 6.23% | 3.35% |
| **Our CoT Gemma-2-9B (fine-tuned)** | **64.71%** | **53.08%** | **32.24%** |
| Llama 3.2-3B (paper fine-tune) | 95.99% | 64.86% | 12.13% |
| Qwen3-4B (paper fine-tune) | 98.04% | 72.42% | 20.19% |
| Mistral 7B (paper fine-tune) | 97.87% | 74.97% | 23.46% |
| GPT-4o (zero-shot) | 41.67% | 32.91% | 16.28% |

**Three key findings:**

1. **SOTA on length generalization.** Our fine-tuned model achieves 32.24% execution accuracy on the length split — **+8.78 pp over the previous best (Mistral 7B at 23.46%), and +12.05 pp over Qwen3-4B (20.19%)**. The length split tests the hardest form of compositional generalization: producing correct queries that are systematically longer than any seen during training. This is the single split where the compositional-reasoning hypothesis is most clearly tested, and where CoT distillation pulls clearly ahead.

2. **Length-degradation gap is much smaller for our model.** The paper baselines drop catastrophically from IID to length: Qwen3-4B falls 77.85 pp (98 → 20), Mistral 7B falls 74.41 pp (98 → 23), Llama 3.2-3B falls 83.86 pp (96 → 12). Our model falls only 32.47 pp (65 → 32). **CoT-distilled training produces a model that degrades gracefully under length generalization rather than collapsing.** This is direct mechanistic evidence for the compositional-reasoning hypothesis: the model has learned to *compose* query primitives, so adding more primitives doesn't break it the way it breaks specialized fine-tunes.

3. **IID and compositional gaps reflect training-data scale, not method ceiling.** Our IID (64.71%) is below the paper's specialized fine-tunes (~98%). This is expected — the paper baselines are trained for more epochs on Pole-specific data, and IID accuracy benefits primarily from memorization of in-distribution patterns. CoT distillation deliberately trades a small amount of IID fit for substantial length-generalization gain, which is the right trade for downstream applications where queries vary in length and complexity.

Combined with §6.7's schema-grounding result, §6.8's latent-vs-active decomposition, and the per-feature taxonomy in §6.9, this is the strongest evidence in the paper that **CoT distillation teaches genuine graph-structural reasoning that generalizes across schemas, syntactic dialects, and query lengths** — not merely memorization of canonical surface forms.

## 7. Qualitative Examples

### 6.1 Example: Exact Match

**Question:** "What are the names of the technicians that have not been assigned to repair machines?"

**CoT Model Output:**
```
Reasoning:
1. Sub-questions: (a) Identify all Technician nodes. (b) Filter out those that have
   an incoming ASSIGNED_TO relationship from any RepairAssignment node.
   (c) Return the remaining technicians' names.
2. Schema elements: Node label Technician with property Name; relationship
   ASSIGNED_TO from RepairAssignment → Technician.
3. Pattern: Simple node match on Technician with NOT EXISTS sub-query filtering
   out those with ASSIGNED_TO.
4. Construction:
   - MATCH (t:Technician)
   - WHERE NOT EXISTS ((:RepairAssignment)-[:ASSIGNED_TO]->(t))
   - RETURN t.Name

Cypher output: MATCH (t:Technician) WHERE NOT EXISTS ((:RepairAssignment)-[:ASSIGNED_TO]->(t)) RETURN t.Name
```

**Reference:** `MATCH (t:Technician) WHERE NOT EXISTS ((:RepairAssignment)-[:ASSIGNED_TO]->(t)) RETURN t.Name`

The reasoning trace correctly identifies the NOT EXISTS pattern, maps to the right schema elements, and produces an exact match.

### 6.2 Example: Semantically Correct, Syntactically Different

**Question:** "Identify the 5 suppliers with the highest average unit price of products supplied."

**CoT Model Output:**
```
Reasoning:
1. Sub-questions: (a) Which suppliers supply products? (b) What is the average
   unitPrice of those products per supplier? (c) Return the top 5 suppliers
   by this average.
2. Schema elements: Supplier node (companyName), Product node (unitPrice),
   relationship SUPPLIES from Supplier to Product.
3. Pattern: Two-hop traversal Supplier-[:SUPPLIES]->Product, then aggregation
   avg(p.unitPrice) grouped by supplier, ordered descending, limited to 5.
4. Construction:
   - MATCH (s:Supplier)-[:SUPPLIES]->(p:Product)
   - WITH s, avg(p.unitPrice) AS avgUnitPrice
   - ORDER BY avgUnitPrice DESC LIMIT 5
   - RETURN s.companyName, avgUnitPrice

Cypher output: MATCH (s:Supplier)-[:SUPPLIES]->(p:Product) WITH s, avg(p.unitPrice) AS avgUnitPrice ORDER BY avgUnitPrice DESC LIMIT 5 RETURN s.companyName AS supplierName, avgUnitPrice
```

**Reference:** `MATCH (s:Supplier)-[:SUPPLIES]->(p:Product) WITH s, avg(p.unitPrice) AS avgUnitPrice ORDER BY avgUnitPrice DESC LIMIT 5 RETURN s.companyName AS Supplier, avgUnitPrice AS AverageUnitPrice`

The query is semantically identical — only the column aliases differ (`supplierName` vs `Supplier`, `avgUnitPrice` vs `AverageUnitPrice`). This would be an execution match but not a string match, illustrating why execution EM exceeds string EM on the DB-accessible subset.

### 6.3 Output Format Compliance

Across all 4,833 test predictions:
- **100%** contain the "Cypher output:" marker
- Average raw output length: 870 characters
- The model consistently follows the trained format: reasoning first, then Cypher

## 10. Limitations and Threats to Validity

### 10.1 Choice of 2024 Dataset Over 2025

We use the `neo4j/text2cypher-2024v1` dataset rather than the cleaner `neo4j/text2cypher-2025v1` for three reasons:

1. **Baseline comparability:** Neo4j published fine-tuned baselines (Gemma-2-9B, Llama-3.1-8B) and reference scores for closed models (GPT-4o, Gemini-1.5-Flash) only on the 2024 dataset. Switching to 2025 would eliminate the leaderboard context that motivates our work.

2. **Concurrent work uses 2024:** Tran et al. (2025) — the most directly comparable RL-based method — also evaluates on 2024. Using the same dataset enables direct comparison of CoT distillation (ours: 0.7682 GLEU) vs. SFT+GRPO (theirs: 0.7701 GLEU).

3. **Known issues do not invalidate relative comparisons:** The 2024 dataset has documented limitations: data leakage (same questions with different Cypher across train/test splits), paraphrase contamination, and same-distribution train/test splits (Ozsoy et al., 2025). These affect both baseline and CoT model equally, so relative deltas are unaffected.

We acknowledge that performance on the 2025 dataset could differ. Future work should validate on 2025 once published baselines exist.

### 10.2 Baseline GLEU Discrepancy

Our reproduced baseline GLEU (0.6455) significantly exceeds Neo4j's published value (0.5560) for the same model under nominally identical conditions. Possible explanations:

- **Input truncation:** 10.5% of test prompts exceed 1,600 tokens. Neo4j's training `max_seq_len=1600` may have applied to inference too; we use `max_length=7,680` preserving full schemas. Examples >1,600 tokens score GLEU 0.5056 vs.\ 0.6622 for ≤1,600 in our setup.
- **Framework versions:** Different transformers, PEFT, or CUDA versions can produce slightly different floating-point behavior.
- **Generation parameters:** Neo4j did not document their exact decoding strategy.

This discrepancy means absolute leaderboard comparisons (mixing our reproduced numbers with Neo4j's published values for other models) are imperfect. We use our own reproduced baseline as the authoritative comparison point: the +0.1227 GLEU delta and +0.0712 execution EM delta are computed under fully identical conditions.

### 10.3 Known Dataset Issues

Beyond the choice of dataset version, the 2024 benchmark has additional limitations:
- **Data leakage:** Same questions may appear with different Cypher outputs across train/test splits
- **Paraphrase contamination:** Paraphrased versions of questions may appear in both splits
- **Distribution shift:** Train and test drawn from the same distribution; real-world performance may differ

### 10.4 Experimental Design Considerations

1. **Two variables changed simultaneously:** The full CoT configuration changes both training data (CoT traces) and inference prompt ("Think step by step"). Our ablation (Section 5) shows that CoT training alone provides +0.0627 GLEU, confirming that the training data is the primary driver.

2. **Single epoch, no hyperparameter tuning:** Following Neo4j's methodology, we train for 1 epoch without tuning. The authors note that "with better-tuned parameters, we could potentially achieve even stronger results." The same applies to our CoT model.

3. **Execution evaluation caveats:** The Neo4j demo databases are live public infrastructure. Database contents may have changed between Neo4j's evaluation and ours.

4. **Reasoning trace quality:** The CoT traces were generated by a single model (GPT-oss-120B). Different reasoning models or prompting strategies might produce different results.

5. **ZOGRASCOPE syntactic mismatch:** The cross-benchmark evaluation tests both schema and syntax transfer. Performance might be higher if the model were fine-tuned on ZOGRASCOPE's inline-WHERE syntax.

## 11. Summary

Chain-of-thought fine-tuning — generating reasoning traces with a strong LLM and training a smaller model on those traces — significantly improves Text2Cypher performance:

- **+0.1227 GLEU** on translation-based evaluation (4,833 test examples)
- **+0.0692 execution EM** on execution-based evaluation (2,471 test examples)
- **53% fewer prediction errors** (114 vs 242 syntax/runtime failures)
- **Improvement across all data sources and 15 of 16 databases**
- **9B open model competitive with fine-tuned GPT-4o-mini** on GLEU (0.7682 vs 0.7973)

The improvement comes primarily from the training data (reasoning traces teach better query construction) with additional gains from the CoT inference prompt (explicit reasoning produces more precise queries). The total cost of the approach — $73 for data generation and training — makes it practical for any research group or organization working with graph databases.
